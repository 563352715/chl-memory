## Why this happened

**Context:** Most of the synthetic-fallback pattern came from the **Emergent dashboard scaffolding cherry-picked into iter 140.x** (commit `e3dc34e` "import Emergent dashboard scaffolding (cherry-pick from conflict_060526_1955)"). Emergent is a no-code platform generator that ships every output with demo data so the UI renders nicely on first run.

**The DB-FIRST-fallback pattern is reasonable for early-stage development** — the platform feels populated even when MongoDB is empty, which is helpful for stakeholder demos and frontend QA.

**Where it goes wrong:**
1. The disclaimer ("Data simulates...") only appears in `/api/carriers` — other endpoints don't even mark fallback data as synthetic.
2. The frontend never surfaces the "demo mode" state to the operator.
3. Pre-revenue solo founder might rationally assume Dashboard's "available drivers: 4" or "loads pipeline: $342K" reflects something real, not a fixture.

**This audit corrects that.** Operator now has a complete picture of what's real vs. demo. Remediation can proceed via #1-#5 above per priority.
